# API client (packages/api-client)

A thin, framework-agnostic typed client. It exports exactly two things: the generated `paths` types
(`schema.d.ts`) and an `openapi-fetch` client created against them. Nothing else.

## Hard boundaries

- **No React, no react-query, no Legend State here.** Those are mobile concerns. The react-query hooks
  (`useRecords`, query keys, cache config) live in `apps/mobile`. Keeping this package UI-free means the
  client could be reused by any TS consumer.
- `src/schema.d.ts` is GENERATED by `openapi-typescript` from the API's OpenAPI document and is gitignored.
  Never hand-edit it. Regenerate with `pnpm gen:api-types` (Turbo task `gen:api-types`).
- The client factory reads the API base URL from the environment of whoever instantiates it (passed in by
  the caller), so it stays environment-agnostic. Export a `createApiClient({ baseUrl })` factory.

## Regen flow

1. Start the API (or have a checked-in `openapi.json`).
2. `pnpm gen:api-types` runs `openapi-typescript <spec> -o src/schema.d.ts`.
3. Types flow into the `openapi-fetch` client; consumers get full path/method/response typing with zero `any`.

If the spec is unreachable during generation, fall back to a committed `openapi.json` snapshot. Keep the
generated types and the running API in sync; a stale `schema.d.ts` defeats the purpose of the pipeline.
